# CineVerse 프로젝트 작업 규칙

## API 응답 형식

이 프로젝트에서 API 응답을 새로 만들거나 수정할 때는 아래 형식을 항상 지킨다.

### 성공 응답

- 최상위에 `state`, `message`를 둔다.
- 반환할 데이터가 있으면 반드시 `data` 안에 넣는다.
- 반환할 데이터가 없는 성공 응답은 `data` 키를 생략할 수 있다.

```python
return {
    "state": "success",
    "message": "처리 성공",
    "data": {
        "id": result.id,
    },
}
```

반환할 데이터가 없는 성공 응답은 다음 형식을 사용한다.

```python
return {
    "state": "success",
    "message": "처리 성공",
}
```

### 정상 실패 응답

- 처리할 대상이 없거나 요청을 수행할 수 없는 경우 `state`는 `failure`를 사용한다.
- 최상위에 `state`, `message`를 둔다.
- 함께 반환할 결과가 있으면 반드시 `data` 안에 넣는다.
- 반환할 데이터가 없는 정상 실패 응답은 `data` 키를 생략할 수 있다.

```python
return {
    "state": "failure",
    "message": "검색 결과가 없습니다.",
    "data": {},
}
```

### 예외 응답

- `except Exception as e:` 형식을 사용한다.
- 최상위에 `state`, `message`, `error`를 둔다.
- 예외 응답에서는 `data`로 오류를 감싸지 않는다.
- 오류 원문은 `error: str(e)` 형식으로 반환한다.
- DB 변경 API라면 예외 응답 전에 `db.rollback()`을 호출한다.

```python
except Exception as e:
    db.rollback()
    return {
        "state": "error",
        "message": "처리 중 에러가 발생했습니다.",
        "error": str(e),
    }
```

## 응답 키 요약

- 성공: `state`, `message`, 필요한 경우 `data`
- 정상 실패: `state`, `message`, 필요한 경우 `data`
- 예외: `state`, `message`, `error`
- 결과 데이터를 `data` 바깥의 임의 키로 반환하지 않는다.
- 성공 또는 정상 실패 응답에서 반환할 데이터가 없으면 `data` 키를 생략할 수 있다.
- 예외 문자열을 `data` 안에 넣지 않는다.

## import 작성 규칙

- 여러 항목을 import할 때 괄호 `()`를 사용해 여러 줄로 나누지 않는다.
- import할 항목은 한 줄에서 쉼표(`,`)로 구분해 작성한다.

```python
from app.models import Movie, Character, User
```

- 다음과 같은 괄호 형식은 사용하지 않는다.

```python
from app.models import (
    Movie,
    Character,
    User,
)
```

## 코드 수정 시 주석 작성 규칙

- 코드를 새로 작성하거나 기존 코드를 수정할 때 관련 로직을 설명하는 한국어 주석을 자세히 작성한다.
- 함수나 메서드를 새로 만들 때는 정의 바로 위에 해당 함수가 어떤 역할을 담당하는지 간략하게 설명하는 한 문장의 한국어 주석을 작성한다.
- 함수 설명 주석은 코드 동작을 그대로 옮기기보다 함수의 핵심 목적이 바로 이해되도록 작성한다.
- 사용자가 직접 수정하지 말고 구현 방법만 알려달라고 요청하거나, 채팅에서 코드 예시·수정 예시·프롬프트 형태로 안내할 때도 예시 코드에 자세한 한국어 주석을 작성한다.
- 채팅으로 코드를 안내할 때는 파일별 역할, 코드를 추가할 위치, 함수의 처리 순서, 입력값과 반환값, 기존 코드와 연결되는 부분을 주석과 설명으로 함께 안내한다.
- 일부 코드만 예시로 보여주는 경우에도 생략된 부분과 사용자가 실제 프로젝트에 맞춰 확인하거나 변경해야 하는 부분을 명확하게 설명한다.
- 함수와 클래스에는 담당 역할, 주요 입력값, 반환값을 이해할 수 있는 설명을 작성한다.
- 인증, 권한 검사, 외부 API 호출, DB 조회·변경, `flush`, `commit`, `rollback`, 감사 로그처럼 중요한 처리에는 목적과 처리 이유를 주석으로 남긴다.
- 여러 테이블을 함께 변경하는 경우 어떤 데이터가 함께 생성·수정·삭제되는지와 하나의 트랜잭션으로 처리하는 이유를 설명한다.
- 예외 처리에는 어떤 상황에서 오류가 발생할 수 있는지와 rollback이 필요한 이유를 설명한다.
- 기존 함수를 재사용할 때는 무엇을 재사용하는지와 중복 구현을 피하는 이유를 필요한 위치에 설명한다.
- 코드 한 줄을 그대로 한국어로 옮기는 주석보다는 해당 코드가 필요한 이유와 전체 흐름에서의 역할을 설명한다.
- 주석은 실제 코드 동작과 항상 일치해야 하며, 로직을 수정할 때 관련 주석도 함께 수정한다.

## 수정 전 안내 및 영향받는 파일 확인 규칙

- 사용자가 수정을 요청하면 실제 작업을 시작하기 전에 어떤 내용을 수정할지 먼저 알린다.
- 사용자가 지정한 파일 외에 다른 파일도 수정하거나 함께 확인해야 하는 경우, 작업 전에 해당 파일과 관여하는 이유를 사용자에게 알린다.
- 작업 도중 다른 파일의 수정 필요성을 새로 발견한 경우에도 해당 파일을 수정하기 전에 사용자에게 먼저 알린다.
- 요청한 변경이 기존 기능, API, DB, 설정 또는 프론트엔드와 연결되어 있으면 영향을 받는 부분을 함께 설명한다.

## 보안 관련 작업 전 안내 규칙

- `.env`, API 키, JWT 비밀키, 비밀번호, DB 접속 정보, 관리자 권한, 인증 토큰, 파일 업로드처럼 보안과 관련된 내용을 수정할 때는 실제 수정 전에 사용자에게 먼저 알린다.
- 수정 전 안내에서는 일반 코드보다 더 자세하고 쉬운 한국어를 사용해 수정 대상과 작업 이유를 설명한다.
- 단순히 변경할 설정값이나 코드만 알리지 않고, 해당 설정이 왜 필요한지, 노출되면 어떤 문제가 생기는지, 개발 환경과 운영 환경에서 어떻게 관리해야 하는지를 함께 설명한다.
- 보안 관련 내용을 직접 수정하지 않고 구현 방법만 안내하는 경우에도 같은 기준으로 설명한다.
- 실제 비밀번호, API 키, 토큰, 비밀키 같은 민감정보를 코드 예시, 주석, 로그, 오류 응답, 문서에 직접 작성하지 않는다.
- 민감정보가 필요한 예시에서는 `your-secret-key`, `your-database-password` 같은 명확한 예시값을 사용하고, 사용자가 자신의 실제 값으로 교체해야 한다는 점을 안내한다.
- `.env` 파일은 Git에 커밋하지 않도록 `.gitignore` 등록 여부를 확인하거나 확인 방법을 안내한다. 다른 개발자에게 필요한 환경변수 목록은 실제 비밀값이 없는 `.env.example`로 공유하도록 안내한다.
- 프론트엔드에 전달되는 값과 서버에서만 보관해야 하는 값을 구분해서 설명한다. 비밀키와 DB 비밀번호처럼 서버 전용인 값은 API 응답이나 프론트엔드 코드에 포함하면 안 된다는 점을 명확히 알린다.
- 로그와 예외 응답에 민감정보가 포함될 가능성이 있으면 어떤 값이 노출될 수 있는지 설명하고, 운영 환경에서는 내부 오류 원문이나 접속정보를 사용자에게 그대로 반환하지 않는 안전한 처리 방법도 함께 안내한다.
- 파일 업로드를 안내할 때는 확장자만 신뢰하지 않고 MIME 타입, 파일 크기, 저장 파일명, 저장 위치를 검증해야 하는 이유와 경로 조작 및 악성 파일 위험을 쉽게 설명한다.
- 보안 관련 변경을 안내할 때는 사용자가 따라 할 수 있도록 적용할 파일과 위치, 설정 예시, 확인 방법, 주의사항을 순서대로 설명한다.
- 이미 민감정보가 Git이나 외부에 노출된 경우에는 파일에서 지우는 것만으로 충분하지 않으며, 해당 키·비밀번호·토큰을 폐기하거나 재발급해야 한다는 점을 반드시 안내한다.
